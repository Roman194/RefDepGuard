# Общая информация
Данное расширение для Visual Studio 2022 позволяет отслеживать изменения зависимостей между проектами и сопоставлять эти зависимости с правилами, заданными в специальном настроечном файле. В случае несоответствия изменений связей заданным настройкам, расширение выдаёт предупреждение о некорректных связях и не даёт сборке завершиться.

Целью данного приложения является контроль связей в решениях и обеспечение их соответствия стандартному виду, принятому в компании. Реализуется в рамках администрирования сборки.

# Принятые обозначения и сокращения
- VS22 (сокр.) - Visual Studio 22 - интегрированная среда разработки (IDE), являющаяся целевой платформой для рассматриваемого расширения. Рассматриваемое расширение является нативным решением для VS22 и имеет множество особенностей реализации, связанных с особенностями реализации этой IDE.
- Решение (Solution) - представляет собой единое пространство проектов (контейнер), которое является единицей VS22 для разделения рабочего пространства "проекта" от всего остального. В рамках VS22 предполагается, что любое рабочее пространство должно иметь ровно одно решение. Одновременный вывод нескольких решений в одном окне в рамках этой IDE не поддерживается.
- Проект (Project) - программа или составная часть программы, выполняющая определённую функциональность. В одном решении может находиться как один, так и множество проектов. Проекты могут имеют связи (референсы) между друг другом.
- Связь/Ссылка/Референс (Reference) - некоторая зависимость проекта от сторонней или системной библиотеки, а также других проектов. В рамках рассматриваемого документа референс и референс между проектами понимаются как синонимичные понятия (т.к. прочие виды связей проекта не рассматриваются).
- TargetFramework - целевая рабочая версия проекта. Обозначает также тип проекта. Пример: netstandard2.0.
- TFM (Target framework monikier) - токен стандартизированного формата, задающий текущую TargetFramework проекта. Пример: netstandard.
- Обязательный референс - референс между проектами, который согласно заданным в файле конфигурации правилам должен обязательно существовать для этого проекта (или нескольких проектов при уровнях Global и Solution).
- Недопустимый референс - референс между проектами, который согласно заданным в файле конфигурации правилам ни в коем случае не должен существовать.
- Файл глобальных конфигураций - Файл, в котором зафиксированы все правила, общие для всех Solution и находящихся в текущей папке.
- Файл конфигураций конкретного решения - Файл, в котором зафиксированы все правила, общие для всех проектов конкретного Solution, а также правила для каждого проекта по отдельности.
- Проблемы (Обнаруженные проблемы) - Представляют собой совокупность ошибок (error) и предупреждений (warning), обнаруженных расширением.
- Правила (Ограничения) - Значения параметров, указанные в файлах конфигурации и влекущие за собой необходимость solution или конкретного проекта соответствовать какому-либо значению (наличие/отсутствие референса, версия TargetFramework).
- Исходящий проект - Проект, из которого "выходит" референс между проектами. Является "выгодополучателем" использования этого референса, так как благодаря нему получает доступ к содержимому входящего проекта. Именно у параметров этого проекта и указывается имя входящего проекта для обозначения этого референса между проектами.
- Входящий проект - Проект, в который "входит" референс между проектами.

## 2. Работа расширения в фоновом режиме
Расширение функционирует не только по команде пользователя, но и в фоновом режиме.
В фоновом режиме оно производит фиксацию текущего состояния связей между проектами и версий TargetFramework проекта, а также проверку соответствия этих параметров ограничениям, указанным в **файле конфигурации**. 
Расширение производит работу в фоновом режиме после стартовой загрузки/открытия решения и во время сборки проекта (решения). В случае обнаружения несоответствия текущих параметров проектов и связей между ними, расширение выводит в "список ошибок" VS22 соответствующие ошибки и предупреждения расширения.
В случае обнаружения ошибок во время сборки, расширение **отменяет сборку** и не даёт ей успешно завершиться до тех пор, пока не будут устранены все найденные расширением ошибки.

При работе в фоновом режиме после загрузки решения расширение также производит подгрузку правил из файлов конфигурации. Поведение расширения при ошибках загрузки информации из файлов конфигурации рассмотрено в пункте 2б</br>

### а) Возможные ошибки и предупреждения расширения

***Ошибки*:**

**Reference error** - Возникает, когда обнаружен референс (связь между проектами), создание которого было запрещено в рамках конфигурационного файла или когда не обнаружен референс, который должен обязательно существовать по требованию из конфигурационного файла.</br>
**Match error** - Возникает, когда на одном уровне (про уровни см. пункт 3в) обнаружено противоречие в правиле референсов: референс одновременно заявлен как обязательный и недопустимый.</br>
**Null property error** - Возникает, когда в конфигурационном файле не было обнаружено одно или несколько из требуемых по шаблону конфигурационного файла свойств (см. пункт 3).</br>
**Framework version comparability error** - Возникает, когда текущая версия TargetFramework проекта превосходит максимально допустимую версию для этого проекта (ограничение в рамках параметра "max_framework_version" конфигурационного файла, см. пункт 3).</br>
**framework_max_version deviant value error** - Возникает, когда в значении параметра "max_framework_version" обнаружено некорректное значение, в том числе когда на уровне Project указан шаблон для ограничений нескольких типов.</br>
**framework_max_version template illegal usage error** - Возникает, когда в значении параметра "max_framework_version" обнаружено некорректное использование шаблона задания различных ограничений для различных типов проектов. Некорректность может заключаться в использовании шаблона на уровне проекта в проекте, имеющем только  один тип TFM или в попытке задать ограничение для TFM, не релевантного для проекта с несколькими различными TFM.</br>

***Предупреждения*:**

**warning** - Безымянное предупреждение возникает в случае, если обнаружено некорректное значение свойства TargetFramework (по мнению расширения) в .csproj проекта. **При корректной работе расширения возникать не должно**.</br>
**Reference Match Warning** - Возникает, когда обнаружено дублирование правила референсов или их противоречие на различных уровнях (то есть когда действует правило "перебивания" логики локальных правил глобальными, см. пункт 3в).</br>
**framework_max_version conflict warning** - Возникает, когда в конфиг-файле значение более глобального параметра "framework_max_version" имеет меньшее значение, чем более локальный; или когда на одном уровне значение супертипа all больше, чем значение одного из типов.</br>
**framework_max_version reference conflict warning** - Возникает, когда обнаружен референс между проектами, среди которых ссылающийся проект имеет меньшее значение "framework_max_version", чем тот, на кого он ссылается (потенциальные ссылки "от меньшего к большему" являются нежелательным поведением, которое может привести к проблемам в будущем, вероятно в дальнейших версиях это предупреждение станет ошибкой).
Также к данному типу предупреждений относятся проблемы, когда текущая версия рефа типа "netstandard" несовместима с текущей версией проекта (согласно его TFM).</br>
**Project not found warning** - Возникает, когда в связи между проектами (по результатам заявленных в файлах конфигураций required_references и (или) unacceptable_references на любом из уровней) обнаружено значение, не соответствующее ни одному из названий существующих проектов решения.</br>
**Project Match Warning** - Возникает, когда в решении обнаружен проект, которого нет в конфиг-файле этого Solution или когда в конфиг-файле проект есть, а в самом решении - нет. Выводится пользователю только если он откажется от обновления/актуализации конфиг-файла при инициализации решения, принудительной фиксации референсов или при запуске сборки решения.</br>
**framework_max_version TFM not found warning** - Возникает, когда в шаблоне задания различных ограничений для различных типов проектов (см. пункт 3А) задан TFM, не соответствующий ни одному из существующих согласно [официальной документации](https://learn.microsoft.com/en-us/dotnet/standard/frameworks). Это может быть вызвано опечаткой или тем, что TFM относится к категории deprecated (согласно официальной документации).</br>
**framework_max_version deviant value warning** - Возникает, когда в значении параметра "max_framework_version" обнаружено несоответствие формату "x.0", где x - любое число.</br>
**Transit references warning** - Выводится, если в решении обнаружены транзитивные связи между проектами и в конфигурационном файле проставлено разрешение на вывод транзитивных связей на всех 3 уровнях (см. пункт 3).</br>

## 3. Работа с конфигурационными файлами
Конфигурационные файлы представляют собой 2 файла с расширением .rdg, в которых фиксируются все правила, которым должны соответствовать проекты и референсы между ними в рассматриваемом Solution.
Для корректного прочтения расширением оба файла должны находиться в корневом каталоге Solution (там же, где лежит соответствующий решению .sln файл).
Их структура представляет собой JSON-файлы, в которых рядом с параметрами записаны значения, по которым эти параметры будут сверяться с фактическим положением дел в загруженном Solution.

_!Важно: конфигурационные файлы загружаются в расширение только при его загрузке, поэтому изменения, внесённые в файлы после этого, не будут учитываться в правилах до загрузки Solution ещё раз!_

Названия этих файлов:
````JSON
"global_config_guard.rdg" и "{название Solution}_config_guard.rdg"
````
**global_config_guard.rdg** представляет собой файл глобальных конфигураций: в нём задаются правила, общие для нескольких Solution. 
Несмотря на то, что VS22 **не поддерживает** одновременную загрузку нескольких Solution в одном окне, всё же возможно разместить 2 и более Solution в одной папке и по отдельности работать с ними, разделяя общие файлы. В таком случае ограничения, указанные в глобальном конфигурационном файле, распространяются на все Solution, лежащие в общей папке.
Пример такого использования:</br>

**{название Solution}_config_guard.rdg** представляет собой файл конфигураций конкретного Solution. В нём указаны все правила, относящиеся к Solution, которое заявлено у него в названии.

### а) Шаблон файла "global_config_guard.rdg" и особенности параметра "framework_max_version":
````JSON
{
	"name":"Global",
	"framework_max_version":"-",
	"report_on_transit_references": false,
	"global_required_references":[
              "Mir.Controller.Model"
        ],
	"global_unacceptable_references":[
               "Mir.Controller.Cfg.Meters.CntCfg"
        ]
}
````
Параметры этого файла:

**name** -  Имя файла. "Global", если рассматривается файл глобальных правил и "{имя_Solution}", если рассматривается файл конфигурации конкретного Solution.

**"framework_max_version"** - Содержит информацию о максимально допустимой версии для параметра **TargetFramework** или его аналога (TargetFrameworkVersion / TargetFrameworks). Данное ограничение вводится вручную. Предполагается, что оно будет использоваться для недопущения повышения версии проекта, повышение которой нежелательно по причине потери поддержки другого проекта или сторонней библиотеки.
Ограничение задаётся в следующем формате:
````JSON
"4.2.7"
````
, где точка (".") - разделитель между цифрами.
 Количество цифр в ограничении не ограничено, но указывать более 3 в большинстве случаев не имеет смысла. 
В случае, если ограничения нет, то указывается прочерк ("-").

_!Важно: в случае, если произвести миграцию проекта с одного типа фреймворка на другой (допустим, с .net framework на .net core), то параметр "framework_max_version" будет иметь некорректное значение, что может повлечь формирование ложных ошибок и предупреждений. Не забудьте при миграции проекта изменить версию этого параметра для мигрируемого проекта!_

Также есть возможность задать **различные ограничения для различных типов проектов**. Для этих целей используется специальный шаблон (формат):

```JSON
"netstandard: 2.1; net: 7.5"
```
, где точка с запятой (";") - разделитель между отдельными ограничениями
двоеточие ("`:") - разделитель между названием типа TargetFramework и максимально допустимой версией в рамках правила.
Пробелы являются незначащими.

_!Важно: указание одного и того же типа max_framework_version ограничения более одного раза в рамках одной строки повлечёт за собой **framework_max_version deviant value error**!_

Шаблон поддерживает **указание в том числе и для одного ограничения**: для этого нужно указать это ограничение без точки с запятой в конце:
```JSON
"netstandard: 2.1"
```

_!Важно: этот шаблон может быть использован на уровне Project только для проектов (про уровни см. пункт 3В), имеющих в .csproj "TargetFrameworks", включающую в себя [РАЗЛИЧНЫЕ типы проектов](https://learn.microsoft.com/en-us/dotnet/standard/frameworks#how-to-specify-a-target-framework) (Допустим, "net" и "netstandard"). Попытка его использования на этом уровне при прочих условиях (в т.ч. и в MAUI-проектах) повлечёт за собой **framework_max_version template illegal usage error**!_

Также в рамках этого шаблона существует поддержка **супертипа all**, обозначающего ограничение **на все** типы проектов. По сути all эквивалентен стандартному формату задания ограничения (просто в стандартном варианте он задаётся неявно), но он также допускает возможность задания более строгого ограничения для конкретных типов проектов:
```JSON
"all: 3.0; net: 2.5"
```
_!Важно: под более строгим ограничением подразумевается ограничение, имеющее меньшую версию, чем версия-ограничение, указанная в супертипе all. Наличие в конкретном ограничении версии, большей чем указана в all, считается противоречием и приводит к появлению **framework_max_version conflict warning**!_

Существует ряд особенностей в использовании типа **.net framework**:

Так как данный тип в более ранних версиях имеет TargetFramework-запись вида "v3.5", а в более поздних - вида "net45", то во избежание конфликтов с проектами типа .NET принято решение об использовании в рамках расширения в качестве имени типа проектов .net framework "**netf**" для всех его версий.
Для установки ограничения для этого типа проектов требуется указать запись следующего типа:
```JSON
"netf: 3.0; net: 7.5"
```
_!Важно: для корректной работы расширения в конфигурационном файле требуется указывать значение версию "netf" через точку!_

**report_on_transit_references** - Указывает, нужно ли выводить предупреждения об обнаруженных транзитивных связях (**Transit references warning**). Имеет в качестве значения по умолчанию false. 

_!Важно: для вывода предупреждений о транзитивных связях для проекта, который имеет такие связи, должно стоять значение true на всех 3 уровнях: global, solution и уровне этого проекта.
В целом логика приоритезации значений **по этому параметру** следующая: Global > Solution > Project!_

**global_required_references** - Содержит информацию о требуемых референсах на глобальном уровне: предполагается, что референсы между проектами, перечисленные в значении этого параметра, должны быть у всех проектов в этой папке (за исключением тех, у которых отключена проверка правил референсов уровня Global и Solution, см. подпункт б). При несоответствии ограничению возникает **Reference error**.

**global_unacceptable_references** - Содержит информацию о референсах, которые запрещены на глобальном уровне для всех проектов рассматриваемой папки (за исключением тех, у которых отключена проверка правил референсов уровня Global и Solution, см. подпункт б). При несоответствии ограничению возникает **Reference error**.

Ограничение задаётся на требуемые и недопустимые референсы задаётся в следующем формате:
```JSON
"Mir.Controller.Main",
"Mir.Controller.Tests.Cfg"
```
, где запятая (",") - разделитель между названиями референсов

### б) Шаблон файла "{название Solution}_config_guard.rdg"
````JSON
{
  "name": "WinFormApp",
  "framework_max_version": "netstandard: 2.0.0; net: 8.0",
  "report_on_transit_references": false,
  "solution_required_references":[
    "Mir.Controller.Main",
    "Mir.Controller.Tests.Cfg"
  ],
  "solution_unacceptable_references":[

  ],
  "projects": {
    "Mir.Controller.Model":{
      "framework_max_version": "3.7.0",
      "report_on_transit_references": false,
      "consider_global_and_solution_references": {
          "required": true,
          "unacceptable": true
      }
      "required_references":[
         "Mir.Controller.Tests",
         "Mir.Controller.Project1"
      ],
      "unacceptable_references":[
        "Mir.Controller.WinProject"
      ]
    },
    "Mir.Controller.ARP56":{
      "framework_max_version": "4.7.2",
      "report_on_transit_references": false,
      "consider_global_and_solution_references": {
          "required": true,
          "unnacceptable": true
      }
      "required_references":[
         "Mir.Controller.Tests",
         "Mir.Controller.Project1"
      ],
      "unacceptable_references":[
         "Mir.Controller.WinProject"
      ]
    }
  }
}

````
Параметры этого файла (те, которые не рассмотрены выше при файле глобальных конфигураций):

**solution_required_references** - По аналогии с global, содержит информацию об обязательных референсах, но только на уровне конкретного решения.

**solution_unacceptable_references** - По аналогии с global, содержит информацию о недопустимых референсах, но только на уровне конкретного решения.

**projects** - Представляет собой множество всех проектов конкретного solution. Внутри содержатся объекты одного шаблона, обозначающие конкретные проекты и ограничения их уровня (project). Каждый объект отличается собственный именем (пример: "Mir.Controller.Model", "Mir.Controller.ARP56").

**consider_global_and_solution_references** - Объект, включающий в себя 2 параметра: "required" и "unacceptable". Эти параметры определяют, действуют ли для текущего проекта ограничения на референсы уровней Global и Solution или нет. Первый отвечает за правила на обязательные референсы, а второй - за правила на недопустимые референсы. Возможные значения для этих параметров: true или false.

_!Важно: если данный параметр не будет обнаружен расширением, то, в отличии от остальных, это не вызовет **Null property error**! Вместо этого объект и его 2 параметра будут учтены как false (аналогичное поведение происходит и с параметром "report_on_transit_references")!_

**required_references** - По аналогии с global, содержит информацию об обязательных референсах, но только на уровне конкретного проекта.

**unacceptable_references** - По аналогии с global, содержит информацию о недопустимых референсах, но только на уровне конкретного проекта.

### в) Различные уровни правил и их приоритизация
Параметры "framework_max_version", "required_references", "unacceptable_references" и им подобные задаются на различных уровнях правил.
Всего в расширении 3 уровня правил:
- **Global** - правила глобального уровня (относятся ко всем проектам текущей папки)
- **Solution** - правила уровня текущего решения (относятся ко всем проектам текущего Solution)
- **Project** - правила текущего проекта (относятся к конкретному проекту)

**Приоритизация правил**:
Правило уровня Project > Правило уровня Solution > Правило уровня Global

_! Исключение: в параметре "report_on_transit_references" используется обратная логика приоритезации (см. пункт 3А)!_

Логика приоритизации правил срабатывает в случае, если обнаружено противоречие между правилами. Так расширение решает, какое правило важнее и должно быть использовано. Пользователю же выводится соответствующее предупреждение об обнаружении противоречия: **framework_max_version conflict warning** или **Match warning** соответственно.

В случае, если **противоречащие правила имеют один уровень**, оба правила не учитываются и выводится соответствующая предупреждение/ошибка: **framework_max_version conflict warning** или **Match error** соответственно.

**Поведение при отсутствии правила**:
Правила для каждого проекта начинают считываться по уровню приоритизации. Таким образом, сначала производится анализ правила уровня Project, затем уровня Solution и, в последнюю очередь, правила уровня Global. 
Если для "framework_max_version" **на первом уровне** обнаружено отсутствие правила ("-"), то производится проверка правила уровня Solution. Если там тоже нет правила, то производится проверка на уровне Global. Если на каком-то из уровней в процессе поиска правила обнаружена ошибка, то выводится **framework_max_version deviant value error** и поиск останавливается.
Для "required_references" и "unacceptable_references" логика несколько иная: правила по уровню приоритизации "наслаиваются" друг на друга и, соответственно, выполняется поочерёдное добавление правил всех уровней, вне зависимости от того, было ли выше обнаружено отсутствие правил. В случае дублирования или противоречия правил более низкоуровневое правило не учитывается и выводится соответствующая ошибка/предупреждение об этом.

### г) Как работает референс между проектами
Проще всего показать как он работает на примере фрагмента графического отчёта (про графический отчёт см. пункт 8):</br>
<img width="405" height="121" alt="image" src="https://github.com/user-attachments/assets/8823cba0-7c98-457b-98ec-1140c6beb4a0" />

Как можно заметить, на рисунке наблюдается референс "WinFormsApp1" -> "WinFormApp".
Если данный референс является обязательным, то в рамках настроек конфигурационных файлов данный референс записывается следующим образом:
```JSON
"WinFormApp":{
      "framework_max_version": "-",
      "report_on_transit_references": false,
      "consider_global_and_solution_references": {
          "required": true,
          "unacceptable": true
      }
      "required_references":[
         "WinFormApp1"
      ],
      "unacceptable_references":[
      ]
    },
```
Таким образом, **в параметрах** "required_references", "unacceptable_references" и т.д. записывается **имя проекта, на который ссылается** референс (обязательный или недопустимый). И, в случае ограничения уровня Project, это ограничение указывается в проекте, имя которого соответствует имени проекта, **с которого** ссылается референс.
Проект с которого ссылается референс считается исходящим, а на которой ссылается - входящим.

**Запрет на референсы "от меньшего к большему"**:
В VS22 реализован запрет для референсов между проектами, который заключается в том, что TargetFramework исходящего проекта **не может быть меньше, чем** TargetFramework входящего проекта.
Это объясняется тем, что в более новых версиях проектов существует поддержка более старых версий, однако, более старые версии ничего не знают про функционал более новых версий. Так как референс между проектами предполагает что исходящий проект получает доступ к функционалу входящего, то он обязательно должен поддерживать весь функционал входящего и, соответственно, иметь версию TargetFramework выше, чем у входящего или равную ему.
Так как данная проверка реализована средствами VS22, то в расширении она не реализована.

Однако, так как в расширении представлено ограничение на макс. значение TargetFramework, то имеет смысл производить проверку на **потенциальный конфликт версий TargetFramework**. 
Таким образом, ограничение на макс. версию TargetFramework исходящего проекта **должно быть больше или равно** аналогичному ограничению для входящего проекта. При несоблюдении этого условия формируется соответствующее предупреждение (**framework_max_version reference conflict warning**).

_!В текущей версии расширения рассматривается ограничение когда обнаружен референс между проектами, среди которых ссылающийся проект имеет меньшее значение "framework_max_version", чем тот, на кого он ссылается и когда текущая версия рефа типа "netstandard" несовместима с текущей версией проекта (согласно его TFM)!_

## 7. Табличный экспорт состояния проекта
При нажатии в меню на кнопку "Экспорт в XLSX" или при нажатии комбинации клавиш Alt + X производится формирование табличного отчёта в формате .xlsx файла.

По результатам экспорта пользователю будет выведен MessageBox с предложением открыть папку с сохранённым экспортом. При нажатии на кнопку "Да" будет произведено описанное ранее действие. Данное сообщение также является сообщение об успешном завершении экспорта.</br>

<img width="434" height="160" alt="image" src="https://github.com/user-attachments/assets/2904ac6b-c00d-40c6-9407-22b1761d3562" />

Файл будет сохранён по следующему пути и будет иметь следующее название:
````JSON
{название_solution}/reports/table_type/{DD.MM.YYYY-HH.MM.SS}/{Название solution}_references_report.xlsx
````
, где D - цифра дня, M - в первом случае цифра месяца, во втором - минуты, Y - цифра года, H - цифра часа, S - цифра - секунды. Вместе данные цифры формируют точную дату и время формирования отчёта, являющиеся уникальным ID отчёта.

Табличный отчёт состоит из трёх страниц:
- Страница выборки по проектам
- Выборки по референсам
- Текущих ошибок RefDepGuard
- Текущих предупреждений RefDepGuard

### а) Страница выборки по проектам
Является представлением всех проектов, входящих в solution по-отдельности.
Рядом с каждым проектом выводятся:
- Его имя
- Целевая рабочая среда (TargetFramework)
- Макс. допустимая версия TargetFramework
- Текущие зафиксированные референсы между проектами (Кол-во и их имена)
- Текущие незафиксированные обязательные референсы между проектами (Кол-во и их имена)
- Текущие зафиксированные недопустимые референсы между проектами (Кол-во и их имена)
В качестве заголовка таблицы выводятся название Solution и дата и время генерации отчёта.</br>

<img width="975" height="263" alt="image" src="https://github.com/user-attachments/assets/60cfba88-3f83-4c15-a6b9-1cbb6bcac6b2" />

В столбце макс. допустимая версия TargetFramework используются следующие условные обозначения:
- S - Solution (ограничение уровня Solution)
- G - Global (ограничение уровня Global)
- [] - Квадратные скобки (указатель на уровень ограничения, отличный от Project)
- ? - Вопросительный знак (пользователь указал ограничение в конфигурационном файле в некорректном формате, сопровождается соответствующей **framework_max_version deviant value error** в списке ошибок)
- Красный цвет ограничения - Обнаружено несоответствие TargetFramework указанному правилу в конфиг-файле
- Жёлтый цвет ограничения - Правило противоречит какому-то другому правилу (того же уровня или различных)
- Прочерк - Не обнаружено никаких правил, создающих ограничение на макс. версию TargetFramework проекта

В отчёте используются следующие обозначения:
- Жирное выделение связи - обязательный референс между проектами
- Красное выделение проекта - проект, нарушающий своё ограничение на макс. версию TargetFramework, указанное в конфигурационном файле
- Жёлтое выделение проекта - проект, имеющий конфликт макс. версии TargetFramework с проектом, с которым он соединён связью
- Красное выделение связи - Либо отсутствующий обязательный референс, либо присутствующий недопустимый. Отличаются названием связи
- Жёлтое выделение связи - Потенциальный конфликт при повышении версии TargetFramework для исходящего проекта, так как обнаружена связь "от меньшего к большему" (см. пункт 2г)

Проекты представляют собой прямоугольники, имеющие 2 - 3 строки. В первой представлено имя проекта, во второй его TargetFramework, а в третьей - ограничение на максимально возможную версию TargetFramework. В случае, **если ограничения на макс. версию нет, то третья строка не выводится**

### а) Приоритизация особых обозначений  при рисовании связей
Так как для одной связи может обнаружится несколько различных особых обозначений, то имеет смысл выводить наиболее важное из них. Приоритизация особых обозначений следующая:

Красное выделение связи > Жёлтое выделение связи > Жирное выделение связи
