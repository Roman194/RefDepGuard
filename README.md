# Общая информация
Данное расширение для Visual Studio 2022 позволяет отслеживать изменения зависимостей между проектами и сопоставлять эти зависимости с правилами, заданными в специальном конфиг-файле. В случае несоответствия изменений связей заданным настройкам, расширение выдаёт предупреждение о некорректных связях и не даёт сборке завершиться.

Целью данного приложения является контроль связей в решениях и обеспечение их соответствия стандартному виду, принятому в компании. Реализуется в рамках администрирования сборки.

## 2. Работа расширения в фоновом режиме
Расширение функционирует не только по команде пользователя, но и в фоновом режиме.
В фоновом режиме оно производит фиксацию текущего состояния связей между проектами и версий TargetFramework проекта, а также проверку соответствия этих параметров ограничениям, указанным в **файле конфигурации**. 
Расширение производит работу в фоновом режиме после стартовой загрузки/открытия решения и во время сборки проекта (решения). В случае обнаружения несоответствия текущих параметров проектов и связей между ними, расширение выводит в "список ошибок" VS22 соответствующие ошибки и предупреждения расширения.
В случае обнаружения ошибок во время сборки, расширение **отменяет сборку** и не даёт ей успешно завершиться до тех пор, пока не будут устранены все найденные расширением ошибки.

### а) Возможные ошибки и предупреждения расширения

***Ошибки*:**

**Reference error** - Возникает, когда обнаружен референс (связь между проектами), создание которого было запрещено в рамках конфигурационного файла или когда не обнаружен референс, который должен обязательно существовать по требованию из конфигурационного файла.</br>
**Match error** - Возникает, когда на одном уровне обнаружено противоречие в правиле референсов: референс одновременно заявлен как обязательный и недопустимый.</br>
**Null property error** - Возникает, когда в конфигурационном файле не было обнаружено одно или несколько из требуемых по шаблону конфигурационного файла свойств.</br>
**Framework version comparability error** - Возникает, когда текущая версия TargetFramework проекта превосходит максимально допустимую версию для этого проекта (ограничение в рамках параметра "max_framework_version" конфигурационного файла).</br>
**framework_max_version deviant value error** - Возникает, когда в значении параметра "max_framework_version" обнаружено некорректное значение, в том числе когда на уровне Project указан шаблон для ограничений нескольких типов.</br>
**framework_max_version template illegal usage error** - Возникает, когда в значении параметра "max_framework_version" обнаружено некорректное использование шаблона задания различных ограничений для различных типов проектов. Некорректность может заключаться в использовании шаблона на уровне проекта в проекте, имеющем только  один тип TFM или в попытке задать ограничение для TFM, не релевантного для проекта с несколькими различными TFM.</br>

***Предупреждения*:**

**warning** - Безымянное предупреждение возникает в случае, если обнаружено некорректное значение свойства TargetFramework (по мнению расширения) в .csproj проекта. **При корректной работе расширения возникать не должно**.</br>
**Reference Match Warning** - Возникает, когда обнаружено дублирование правила референсов или их противоречие на различных уровнях (то есть когда действует правило "перебивания" логики локальных правил глобальными.</br>
**framework_max_version conflict warning** - Возникает, когда в конфиг-файле значение более глобального параметра "framework_max_version" имеет меньшее значение, чем более локальный; или когда на одном уровне значение супертипа all больше, чем значение одного из типов.</br>
**framework_max_version reference conflict warning** - Возникает, когда обнаружен референс между проектами, среди которых ссылающийся проект имеет меньшее значение "framework_max_version", чем тот, на кого он ссылается (потенциальные ссылки "от меньшего к большему" являются нежелательным поведением, которое может привести к проблемам в будущем, вероятно в дальнейших версиях это предупреждение станет ошибкой).
Также к данному типу предупреждений относятся проблемы, когда текущая версия рефа типа "netstandard" несовместима с текущей версией проекта (согласно его TFM).</br>
**Project not found warning** - Возникает, когда в связи между проектами (по результатам заявленных в файлах конфигураций required_references и (или) unacceptable_references на любом из уровней) обнаружено значение, не соответствующее ни одному из названий существующих проектов решения.</br>
**Project Match Warning** - Возникает, когда в решении обнаружен проект, которого нет в конфиг-файле этого Solution или когда в конфиг-файле проект есть, а в самом решении - нет. Выводится пользователю только если он откажется от обновления/актуализации конфиг-файла при инициализации решения, принудительной фиксации референсов или при запуске сборки решения.</br>
**framework_max_version TFM not found warning** - Возникает, когда в шаблоне задания различных ограничений для различных типов проектов задан TFM, не соответствующий ни одному из существующих согласно [официальной документации](https://learn.microsoft.com/en-us/dotnet/standard/frameworks). Это может быть вызвано опечаткой или тем, что TFM относится к категории deprecated (согласно официальной документации).</br>
**framework_max_version deviant value warning** - Возникает, когда в значении параметра "max_framework_version" обнаружено несоответствие формату "x.0", где x - любое число.</br>
**Transit references warning** - Выводится, если в решении обнаружены транзитивные связи между проектами и в конфигурационном файле проставлено разрешение на вывод транзитивных связей на всех 3-х уровнях.</br>

## 3. Работа с конфигурационными файлами
Конфигурационные файлы представляют собой 2 файла с расширением .rdg, в которых фиксируются все правила, которым должны соответствовать проекты и референсы между ними в рассматриваемом Solution.
Для корректного прочтения расширением оба файла должны находиться в корневом каталоге Solution (там же, где лежит соответствующий решению .sln файл).
Их структура представляет собой JSON-файлы, в которых рядом с параметрами записаны значения, по которым эти параметры будут сверяться с фактическим положением дел в загруженном Solution.

Названия этих файлов:
````JSON
"global_config_guard.rdg" и "{название Solution}_config_guard.rdg"
````
**global_config_guard.rdg** представляет собой файл глобальных конфигураций: в нём задаются правила, общие для нескольких Solution. 
Несмотря на то, что VS22 **не поддерживает** одновременную загрузку нескольких Solution в одном окне, всё же возможно разместить 2 и более Solution в одной папке и по отдельности работать с ними, разделяя общие файлы. В таком случае ограничения, указанные в глобальном конфигурационном файле, распространяются на все Solution, лежащие в общей папке.
Пример такого использования:</br>

**{название Solution}_config_guard.rdg** представляет собой файл конфигураций конкретного Solution. В нём указаны все правила, относящиеся к Solution, которое заявлено у него в названии.

### в) Различные уровни правил и их приоритизация
Параметры "framework_max_version", "required_references", "unacceptable_references" и им подобные задаются на различных уровнях правил.
Всего в расширении 3 уровня правил:
- **Global** - правила глобального уровня (относятся ко всем проектам текущей папки)
- **Solution** - правила уровня текущего решения (относятся ко всем проектам текущего Solution)
- **Project** - правила текущего проекта (относятся к конкретному проекту)

**Приоритизация правил**:
Правило уровня Project > Правило уровня Solution > Правило уровня Global

_! Исключение: в параметре "report_on_transit_references" используется обратная логика приоритезации!_

Логика приоритизации правил срабатывает в случае, если обнаружено противоречие между правилами. Так расширение решает, какое правило важнее и должно быть использовано. Пользователю же выводится соответствующее предупреждение об обнаружении противоречия: **framework_max_version conflict warning** или **Match warning** соответственно.

В случае, если **противоречащие правила имеют один уровень**, оба правила не учитываются и выводится соответствующая предупреждение/ошибка: **framework_max_version conflict warning** или **Match error** соответственно.

### г) Способы создания файлов конфигурации
Вы можете создать файлы конфигурации одним из следующих способов:

- Воспользовавшись автогенерацией шаблона расширением при необнаружении файла в корневой директории Solution (рекомендуется)

То есть референсов, на которые ссылается проект без посредничества каких-либо проектов. Если у какого-то из проектов отсутствуют референсы на другие проекты, то ставится прочерк ("-").</br>

При нажатии в меню на кнопку "Зафиксировать текущую версию решения" или при нажатии комбинации клавиш Alt + C производится фиксация текущих референсов между проектами, текущих TargetFramework версий проекта и проверка соответствия этих текущих параметров правилам, загруженным из файлов конфигурации.
По завершении выполнения проверки выводится либо список найденных расширением проблем, либо предупреждение об отсутствии референсов между проектами, либо сообщение о том, что проблемы в решении не были обнаружены.

## 7. Табличный экспорт состояния проекта
При нажатии в меню на кнопку "Экспорт в XLSX" или при нажатии комбинации клавиш Alt + X производится формирование табличного отчёта в формате .xlsx файла.

По результатам экспорта пользователю будет выведен MessageBox с предложением открыть папку с сохранённым экспортом.</br>

<img width="434" height="160" alt="image" src="https://github.com/user-attachments/assets/2904ac6b-c00d-40c6-9407-22b1761d3562" />

Файл будет сохранён по следующему пути и будет иметь следующее название:
````JSON
{название_solution}/reports/table_type/{DD.MM.YYYY-HH.MM.SS}/{Название solution}_references_report.xlsx
````
Табличный отчёт состоит из трёх страниц:
- Страница выборки по проектам
- Выборки по референсам
- Текущих ошибок RefDepGuard
- Текущих предупреждений RefDepGuard
